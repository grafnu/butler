# Instructions for Static Merge Analysis & Specification Refinement (MERGER.md)

This file governs the **Specification Merge** phase of the development lifecycle. This phase is executed strictly *after* a cross-implementation verification run (via `bin/run_cross`) has been completed. 

The primary goal of this phase is to consume the static evidence generated by the cross run, analyze implementation behaviors and discrepancies, resolve specification ambiguities or gaps, and refine the formal specifications.

---

## 1. Prerequisites and Sequence

The spec-driven development and verification pipeline operates in a strict sequence:
1. **bin/run_updates**: Automatically builds and synchronizes all implementation branches with the core specification.
2. **bin/run_cross**: Executes the live cross-implementation test matrix, generating static execution logs, test summaries, and diagnostics.
3. **bin/run_merger** (this file): Performs a purely static, offline analysis of the outputs from the cross run, updating specifications and guidelines in place.

---

## 2. Merger Agent Static Analysis Role & Core Goals

The role of the Merger Agent (governed by this `MERGER.md` skill) is strictly analytical and static. It operates after the cross-test run has completed.

### Core Directives:
1. **NO Live Execution:** The Merger Agent **MUST NOT** execute, compile, build, start, or run any of the implementations, local brokers, or live testing harnesses. It is a purely offline static analysis phase.
2. **NO Push to Git:** Under no circumstances should the Merger Agent push any changes to git. All modifications and proposals must be left in the active working branch for human review and validation.
3. **Static Evidence Analysis:** The Merger Agent must consume and analyze:
   - Recent git changes/diffs across the workspace.
   - Already existing static log files (such as `impl/*.log` or `out/*.log` generated by `bin/run_cross`).
   - Already existing test summary files (specifically `impl/test_summary.txt`).
   - Already existing feedback files (such as `_analysis.md` files, `spec_analysis.md`, or `merge_analysis.md` in implementation directories).
4. **Spec & Skill Evolution:** The primary output of the Merger Agent is updating the `.md` specification files (under `spec/`, e.g., `spec/butler.md`) and core workspace skill/configuration files (such as `UPDATE.md`) to resolve ambiguities, contradictions, or gaps highlighted by the static analysis.

---

## 3. Systematic Analysis & Specification Refinement Workflow

The Merger Agent must perform the static analysis and refinement of the workspace files through the following steps:

### Step 1: Audit Static Input Sources
- Read and inspect `impl/test_summary.txt` to identify which cross-implementation pairs succeeded or failed (`PASS`/`FAIL`).
- Locate and read the relevant log files (e.g., `impl/logs/*.log`, `out/*.log`) for any failing pairs to pinpoint the behavioral discrepancies.
- Check each implementation directory (e.g., `impl/{ID}/`) for any `spec_analysis.md` files generated by the Implementation Agents. These files describe precise spec ambiguities or incompatibilities.
- Review recent git diffs across implementation folders to see how individual implementations resolved issues.

### Step 2: Identify Ambiguities and Contradictions
Analyze the collected static data against the existing specifications in the `spec/` directory. Look for:
- Gaps in the specification that allowed implementations to diverge behaviorally.
- Redundant or multi-path requirements where a single, unified standard is required.
- Incompatible assumptions about data payloads, handshakes, or message topic formatting.

### Step 3: Update Specifications, Skill Files, and Propose Upstream Fixes
Translate the analytical findings directly into explicit, rigorous, and authoritative requirements:
- **Upstream Spec Proposals:** If the root cause of an ambiguity or contradiction stems from generic or incomplete language in the foundational UUFI specification (`impl/udmi/docs/specs/uufi.md`), the Merger Agent MUST clearly document what specifically needs to change in `uufi.md` (e.g., standardizing handshake payloads, defining topic suffix rules, or restricting expected version schemas). **Do not attempt to modify the `uufi.md` file directly**, as the `impl/udmi` repository is immutable in this workspace. Instead, write these proposals clearly into a dedicated `uufi_analysis.md` file at the root of the parent workspace to cleanly separate them from local `butler.md` proposed changes.
- **Local Spec Updates:** Update the local `.md` files under `spec/` (e.g., `spec/butler.md`) ONLY to clarify rules that are strictly specific to the Butler application logic and do not overlap with the broader UUFI transport specification. Do not soften requirements or create redundant rules.
- **Skill Updates:** Update `UPDATE.md` to guide the Implementation Agents on how to handle the clarified rules and avoid repeating the same non-compliance patterns.

## 4. Core Specification Integrity Guidelines

When updating the specifications, adhere strictly to these critical interoperability standards:
1. **Handshake Payload Formatting:** Mandate exactly ONE standard flattened format for Handshake Step 1 and Step 2 request/reply blocks, where the `"setup"` and `"reply"` payload blocks reside directly at the payload root (no nested `"udmi"` wrappers).
2. **Model Updates Structure:** Mandate exactly ONE standard format for cloud model update payloads, where the `"registries"` key resides directly at the payload root (no nested `"cloud"` wrappers).
3. **Expected Version Configuration:** Specify exactly ONE standard way to define the expected version: under the standard software dictionary structure within the device's system configuration (`system.software.<subsystem>`). Any alternative properties like `"target_version"` are strictly prohibited and MUST NOT be accepted.
4. **Topic Suffix Formatting:** Clarify that all UUFI topic paths MUST include both a subtype and a subfolder segment (`/c/{subtype}/{subfolder}`), and topic generation utilities must not generate truncated or omitted suffixes.
5. **Subsystem Reporting Structure:** Ensure the actual state reporting matches standard UDMI schemas, wrapping the subsystem ID (specifically `"system"`) within the `"blobs"` dictionary under `"blobset"`.
6. **Envelope and Configuration Attributes:** Specify compliance rules for envelope attributes like `"nonce"` deduplication/graceful-processing, and config attributes like `"version"`.

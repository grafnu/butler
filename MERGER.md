# Instructions on how to run a merge integration test between multiple implementations.

---

## Coordinated Spec-First Agentic Workflow & Separation of Responsibilities

This project operates on a strict **spec-first development and testing** paradigm, where the primary objective is to evolve the core Butler specification into a robust, complete, and unambiguous definition of system behavior, rather than simply patching individual implementations.

To achieve this, the system defines a highly coordinated agentic workflow with a clean separation of responsibilities between two specialized roles:

1. **Verify & Update Spec (The Merger Agent - governed by `MERGER.md`):**
   - **Responsibility:** Strictly verifies spec-compliance across all cross-implementation pairs by executing integration tests. Does NOT make any code modifications or attempt to fix implementations.
   - **Feedback Loop to Implementations:** If an implementation is at fault, the Merger Agent generates a `merge_analysis.txt` file in the root of that implementation's repository describing the exact failures. If the implementation is correct, any existing `merge_analysis.txt` is removed.
   - **Consuming Spec Feedback:** The Merger Agent reviews any `spec_analysis.txt` files generated by the Implementation Agents (which document ambiguities, contradictions, or external incompatibilities).
   - **Spec Evolution:** If there is any ambiguity, contradiction, or omission in the Butler spec itself, or if a `spec_analysis.txt` suggests a streamlining recommendation, the Merger Agent makes recommended changes to the specifications under the `spec/` directory or modifies the `UPDATE.md` skill on the merger branch to resolve the issue cleanly and elegantly.
   - **Artifacts:** Only specification/skill changes (under `spec/` or `UPDATE.md`), the implementation feedback (`merge_analysis.txt` additions/deletions), and the resolution of `spec_analysis.txt` files are committed and pushed.

2. **Implement Spec (The Implementation Agent - governed by `UPDATE.md`):**
   - **Responsibility:** Responsible for modifying code inside its specific implementation directory (`butler/` and `bin/`) to satisfy the specification, resolving identified non-compliance, and raising spec-level feedback.
   - **Execution Loop:** Upon checkout, the Implementation Agent merges in the latest specs from `main`, checks for `merge_analysis.txt`, and audits the specification files under `spec/`.
   - **Feedback Loop to Spec:** If there are specific ambiguities, contradictions, omissions, or external spec (e.g. UUFI, libraries) incompatibilities/inconsistencies that prevent creating a clear, correct implementation, the Implementation Agent MUST create or update a `spec_analysis.txt` file at the root of the repository. This file must be highly precise, describing clear, non-speculative problems and providing elegant recommendations to streamline the spec (aiming for an elegant, single standard way of doing things, avoiding workarounds or slop). Performance improvements, speculative ideas, or security enhancements should NOT be listed. If no such obstacles exist and the spec is clear and correct, any existing `spec_analysis.txt` file MUST be removed.
   - **Code Refinement:** The agent refines the implementation logic until all requirements in `spec/` and all issues reported in `merge_analysis.txt` are fully resolved and local smoke tests pass. Once verified, `merge_analysis.txt` is deleted, and the updated implementation is committed and pushed.
   - **Spec Integrity:** The Implementation Agent never modifies specifications or files outside of the allowed directories (`butler/`, `bin/`, and `UPDATE.md` if directed).

Through this continuous loop of verification and feedback (Merger Agent) and refinement and spec-auditing (Implementation Agent), the Butler specification is iteratively audited, streamlined, and hardened, achieving true specification-driven robustness.

The merger branch (typically `merger`) can commit and push changes to the `spec/` directory and/or the `UPDATE.md` skill file. No code changes should be made to any implementation code under `impl/` or other directories.

For each sibling implementation branch (`impl_{ID}`):
- If the implementation is deemed to be at fault (failed tests or spec non-compliance), a `merge_analysis.txt` file must be created or updated at the root of that implementation's repository, providing a detailed summary of why the implementation is bad or non-compliant so a subsequent agent can diagnose and fix it properly.
- If the implementation is deemed to be correct and fully compliant (passing all tests), any existing `merge_analysis.txt` file in its repository must be removed.
- If the Implementation Agent created, updated, or removed a `spec_analysis.txt` file on the branch (describing specification ambiguities or external incompatibilities), that file should also be part of the branch.
- All implementation branches with additions or deletions of `merge_analysis.txt` or `spec_analysis.txt` must be committed and pushed back to the remote origin if permissions allow.

If there are changes to the specifications or implementation guidelines on the merger branch, a summary commit detailing those changes must be created and pushed. If there are no changes, an empty commit with the exact commit message "Clean test run" must be pushed to the merger branch. A commit must be created in all cases.

The primary goal is to empirically ensure that the provided specs are robust, coherent, and comply to the spec, by interoperability testing between multiple different implementations of the `butler` spec. Spec compliance is defined as successfully passing the `smokeit` test across all cross-implementation pairs with zero verification failures and consistent behavioral logs.

The goal is to be spec compliant, not to just pass the tests. The spec should be concise and authoritative. It should focus on the specific requirements, avoiding "should" and "may", focusing on what is strictly required, what the system must do to function and be compliant. Either something is done correctly, or it doesn't matter how it's done.

The merger agent MUST NOT attempt to fix or patch implementation code directly. Instead, the agent runs the integration tests to evaluate implementations. If there is an ambiguity, contradiction, or omission in the Butler spec itself, then the agent should make recommended changes in the `spec/` directory or `UPDATE.md` skill of the merger branch for the implementation agent to fix the problem. No code changes should be made to any implementations.

First merge `origin/main` into this branch to make sure all specs and other details are up to date.

There are (should be) multiple working subdirectories, each one containing a different version of an implementation. Check the remote branches of the form `impl_ID`. Each one is a different implementation of the butler spec.

**Cloning and Syncing Branches & Parallel Fetch/Audits:**
To prepare the sibling implementations for testing, list and clone/sync them. To minimize tool and test initialization latency, the upstream synchronization (fetching/cloning) and sibling directory structure audits MUST be executed in parallel (concurrently) across all discovered implementation branches rather than sequentially. This requires:
- Retrieving the list of remote implementation branches (branches prefixed with `impl_`).
- Concurrently cloning new implementation repositories or fetching updates and performing a hard reset on existing directories in the background.
- Concurrently verifying the presence of required file resources (such as `AGENTS.md`) across all synchronized implementation workspaces, failing immediately if the workspace structure is non-compliant.

**Setting up Independent Virtual Environments:**
Each implementation must run within its own isolated Python virtual environment based on its specific `requirements.txt`. Establish, activate, and satisfy all requirements using the Python packages installer prior to setting up the testing environments or executing test suites.

Execute the functional equivalent of the `smokeit` integration tests using the appropriate tools from different subdirectories. Two versions are tested at a time, one primarily for `verifier` the other for `butler`. Both implementations should use the same connectivity specification (as defined in `AGENTS.md`) to ensure they can communicate.

Run and capture the output/logs of both the `butler` and `verifier` processes to separate log files (`impl/{ID}.log`).

**Specification Audit and Update Guidelines:**
The specification MUST be strict, authoritative, clear, and unambiguous. It must NOT be relaxed or updated just to accommodate non-compliant behaviors of buggy test frameworks or sub-implementations. If a test harness (like a cross-testing monitor) or any system implementation is found to use a non-compliant or deviant format (such as nested payload wrappers or custom version fields), the test harness or implementation itself MUST be failed and corrected to comply with the authoritative specification, rather than modifying the specification to allow non-standard alternatives.

Ensure that the specifications clearly define strict, unique, and unambiguous requirements for the following critical interoperability areas:
1.  **Handshake Payload Formatting:** Mandate exactly ONE standard flattened format for Handshake Step 1 and Step 2 request/reply blocks, where the `"setup"` and `"reply"` payload blocks reside directly at the payload root (no nested `"udmi"` wrappers).
2.  **Model Updates Structure:** Mandate exactly ONE standard format for cloud model update payloads, where the `"registries"` key resides directly at the payload root (no nested `"cloud"` wrappers).
3.  **Expected Version Configuration:** Specify exactly ONE standard way to define the expected version: under the standard software dictionary structure within the device's system configuration (`system.software.<subsystem>`). Any alternative properties like `"target_version"` are strictly prohibited and MUST NOT be accepted.
4.  **Topic Suffix Formatting:** Clarify that all UUFI topic paths MUST include both a subtype and a subfolder segment (`/c/{subtype}/{subfolder}`), and topic generation utilities must not generate truncated or omitted suffixes.
5.  **Subsystem Reporting Structure:** Ensure the actual state reporting matches standard UDMI schemas, wrapping the subsystem ID (specifically `"system"`) within the `"blobs"` dictionary under `"blobset"`.
6.  **Envelope and Configuration Attributes:** Specify compliance rules for envelope attributes like `"nonce"` deduplication/graceful-processing, and config attributes like `"version"`.

**Fail-Fast on UDMI Tooling Problems & Zero Patching/Hacks:**
- To preserve the integrity and predictability of the testing framework, there MUST be absolutely no patching, hacking, modifying, or working around of any tooling, scripts, or utilities provided by UDMI (such as `start_mosquitto`, `start_udmis`, `bin/container`, `bin/build`, `mosquitto_ctrl.sh`, etc.) during runtime or in the test harness.
- If any problem, failure, syntax error, missing resource, certificate issue, or misconfiguration is identified with the UDMI tooling itself, the test harness MUST immediately abort, flag the error, and fail-fast. No dynamic runtime patching of UDMI files is allowed under any circumstances.
- **Prohibition of `UDMI_NO_SUDO` Environment Variable:** To ensure realistic validation of the system's automatic no-sudo user-space execution paths, the test harness, runner scripts, and supporting tools MUST NOT explicitly set, export, or pass the `UDMI_NO_SUDO` environment variable during testing. This variable is an internal-only implementation detail of UDMI. All tools under test must automatically determine and run in no-sudo mode when a dynamic port is specified, and if `sudo` is still called, the run must fail-fast as a fundamental compatibility error.
- **System Package Integrity & No Library-Path Hijacking:** All background service daemons (such as `mosquitto`, `etcd`, `java`) and tools under test MUST rely strictly on the standard system libraries and configurations natively provided by the host OS or container environment. Modifying, overriding, or prepending directory paths to environment variables like `LD_LIBRARY_PATH`, `DYLD_LIBRARY_PATH`, or `PATH` to inject loose, stashed, or dynamic shared libraries (such as `.so`, `.dylib`, or `.dll` files) from within the repository is strictly PROHIBITED. If an executable or dependency fails to load due to a missing system library, the test runner or agent MUST NOT attempt to dynamically resolve it using local library stuffing or environment hacks. Instead, it MUST immediately fail-fast and report that the host environment lacks the required system packages, allowing the underlying environment to be fixed properly.
- **Strict Non-Interactive Fail-Fast Environment:** To prevent the execution from hanging indefinitely on unresolved prompts, all tools, setups, test runners, and background commands MUST be executed in a strictly non-interactive environment with Standard Input (`stdin`) closed or redirected to `/dev/null` (e.g., `cmd < /dev/null`). All dynamic interactive prompts, user selections, and administrative password prompts (such as `sudo` credentials, `gcloud` confirmation questions, or interactive input `read` scripts) MUST be configured to fail-fast. Under no circumstances should custom mock scripts or interactive loops be used to work around missing inputs. Any requirement for manual interactive intervention MUST be treated as a critical compatibility error and immediately terminate with a non-zero exit status.
- **Predictable Workspace Scaffolding Mandate:** The directory structure of the testing workspace is fixed, immutable, and strictly defined. The agent MUST NOT execute generic directory scans, recursive finds, or path explorations to determine the layout. Every sibling implementation directory `impl/{ID}/` MUST conform to this exact relative layout: local setup script must be at `impl/{ID}/bin/setup`, local test execution script must be at `impl/{ID}/bin/smokeit`, local immutable UDMI dependency must be at `impl/{ID}/impl/udmi/`, and local isolated site model must be at `impl/{ID}/testing/udmi_site_model/`. Any deviation from this structure is a critical structural violation. If any path is missing or non-compliant, the test runner MUST immediately hard-fail.
- **Opaque "Black-Box" Tooling Rule:** All dependencies, scripts, and utilities inside the immutable `impl/udmi` repository MUST be treated strictly as black boxes. Opening, parsing, reading, or analyzing the source code of any UDMI utility (such as `start_local`, `start_mosquitto`, `start_udmis`, `setup_ca`, etc.) to understand or debug its internal logic is strictly PROHIBITED. The tools must either run successfully according to their command-line interfaces or fail. If a tool crashes or behaves unexpectedly, the agent MUST NOT attempt to diagnose, trace, or work around the internal script logic; it MUST immediately declare a tooling error and hard-fail.
- **Opaque Setup Responsibility:** The test harness MUST NOT perform manual environment preparation steps on behalf of the implementations, such as copying files, creating symbolic links, stashing certificates, or configuring dynamic port overrides. All environment preparation, cloning of internal site models, virtual environment activation, and local service boots MUST be handled natively and opaque-ly inside the implementation's own `bin/setup` script. The harness simply invokes `bin/setup <conn_spec>` under the fail-fast timeout, and if the setup does not exit with code `0`, it is a hard test failure.
- The absolute primary goal is to audit and update the specifications under the `spec/` directory to define a robust, stable, and spec-compliant system. Completing and passing the interoperability tests is a strictly secondary task and MUST NOT be achieved through hacks, workarounds, or patching of the testing tools. If the spec or tool is flawed, it must fail cleanly and be corrected at the spec level or reported as a failure.

If the smoke test fails:
1. Diagnose and analyze whether the failure is due to an implementation fault or an issue in the specification itself (ambiguity, contradiction, or omission).
2. For implementation faults, do NOT modify the implementation code. Create/update `merge_analysis.txt` in that implementation's repository with a comprehensive diagnostic report.
3. For specification issues (ambiguities, contradictions, or omissions), make recommended changes in the `spec/` directory or update the `UPDATE.md` skill file of the merger branch for the implementation agent to fix the problem. No implementation code changes are made.

If the smoke test passes, analyze the generated log files to see if there is any other discrepancy or ambiguities that should be addressed in the spec or `UPDATE.md` skill, and likewise update them.

**Dynamic Port Allocation for Parallel Runs:**
To prevent parallel tests from conflicting and causing socket bind/port collisions, you MUST dynamically allocate a unique free port for each parallel broker instance. Implementations MUST programmatically obtain a free, unoccupied local TCP port from the OS at runtime, and pass this unique port configuration to all setup and verification runners.

**Implementation-Level Parallel Tracks (No Sibling-Run Contention):**
- Parallel execution MUST NOT be structured by simple thread counts of all combined pairs. Running multiple target tests under the same verifier implementation in parallel (such as `impl_A verifies impl_B` and `impl_A verifies impl_C` concurrently) leads to severe resource contention, shared directory file writes, and port conflicts.
- Instead, the parallel execution MUST be structured cleanly by Verifier Implementation:
  1. Exactly **one parallel track** is launched per Verifier implementation (e.g., if there are four implementations, exactly 4 parallel tracks are active at any time).
  2. Within each parallel track, the verifier implementation is tested **sequentially** against each of the other target sibling implementations (e.g., track A runs `impl_A verifies impl_B`, then `impl_A verifies impl_C`, then `impl_A verifies impl_D` sequentially; while in parallel, track B runs `impl_B verifies impl_A`, then `impl_B verifies impl_C`, then `impl_B verifies impl_D` sequentially).
- **Hashed Port Allocation Strategy Compatibility:** To align with systematic port hashing, each parallel track's port mapping MUST be mathematically unique. This is guaranteed by using the **Verifier Implementation Identifier** (e.g., `"impl_A"`, `"impl_B"`, etc.) as the core input string for the SHA256 cryptographic port calculation. Because each parallel track maps to exactly one Verifier, and all sibling verifications (verifiees) within that track run sequentially, every parallel track is guaranteed a unique, collision-free hashed port mapping.
- This approach completely isolates the runtime workspace and port allocations of each Verifier implementation, preventing any port or file conflicts and keeping the peak JVM and broker concurrency strictly capped by the number of implementations. All trial runs should take about the same amount of time to complete, so if some runs are taking more than twice as long as the passing runs, then it should be aborted and considered a failure. If parallel execution hits environment-level timeouts or command-tracking limitations, sequential execution may be used as a robust diagnostic fallback.

**Safe Process Cleanup & Anti-Self-Killing Mandates:**
To guarantee safe operation in shared host or containerized environments, the agent MUST adhere to strict, non-negotiable process cleanup boundaries to prevent accidental termination of the encapsulating agentic `gemini` environment, CLI shell, or other unrelated system services:
- **Absolute Prohibition of Indiscriminate/Sweeping Killing:** Under no circumstances should any agent, script, test runner, or cleanup routine invoke sweeping process termination commands such as `pkill`, `killall`, or broad user-wide/system-wide process signals (e.g., `pkill -u $USER`, `pkill -u <username>`, `killall -u $USER`, `pkill java`, `pkill mosquitto`, `pkill etcd`). Such broad commands kill too many things outside of the isolated test directory (including the `gemini` process itself), leading to immediate tool failure and agent crash.
- **Strict Self-Preservation:** Process signaling or group-based killing (such as `kill 0`, `kill -9 -1`, or sending signals to the agent's own process group/PID) is strictly forbidden. The agent must never run commands that can terminate its own shell, process, or parent process tree.
- **Mandatory Port-Based or PID-Based Targeting:** When clearing lingering background dependencies (such as mosquitto, etcd, or Java pubber processes) from prior runs, the agent MUST target ONLY those processes that are specifically associated with the test execution. This must be accomplished using one of the following precise methods:
  1. *Port-Based Querying:* Query for the specific process ID listening on the dynamically-allocated test port `PORT` using standard tools (e.g., `lsof -t -i :$PORT` or `ss -lptn 'sport = :$PORT'`) and kill ONLY those targeted PIDs.
  2. *Highly-Specific Path Filtering:* If fallback port-free discovery is necessary, use `pgrep -f` with highly specific, non-ambiguous patterns matching the exact sandbox directory path or test identifier (e.g., `pgrep -f "mosquitto.*impl/"` or `pgrep -f "java.*pubber.*impl/"`) rather than generic executable names.
  3. *PID Tracking:* Track and store the exact PIDs of all background-spawned subprocesses in variables, and explicitly kill only those recorded PIDs during cleanup.

PASS means the trial run passed unmodified, and FAIL means it failed and could not be verified successfully without resolving the implementation fault or specification contradiction.

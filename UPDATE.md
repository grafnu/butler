# Update Procedure for Butler Implementation Builds

DO NOT ASK FOR CLARIFICATION OR DIRECTION. YOU ARE AN AUTONOMOUS AGENT WITHOUT
THE ABILITY TO ASK QUESTIONS OR SEEK ADVICE. YOU SHOULD ALWAYS PROCEED AND OPEN
THE PR.

This document outlines the standard procedure for pulling in specifications from
`main` and updating implementation builds to ensure full compliance and stability.

---

## Coordinated Spec-First Agentic Workflow & Separation of Responsibilities

This project operates on a strict **spec-first development and testing** paradigm, where the primary objective is to evolve the core Butler specification into a robust, complete, and unambiguous definition of system behavior, rather than simply patching individual implementations.

To achieve this, the system defines a highly coordinated agentic workflow with a clean separation of responsibilities between two specialized roles:

1. **Verify & Update Spec (The Merger Agent - governed by `MERGER.md`):**
   - **Responsibility:** Strictly verifies spec-compliance across all cross-implementation pairs by executing integration tests. Does NOT make any code modifications or attempt to fix implementations.
   - **Feedback Loop to Implementations:** If an implementation is at fault, the Merger Agent generates a `merge_analysis.txt` file in the root of that implementation's repository describing the exact failures. If the implementation is correct, any existing `merge_analysis.txt` is removed.
   - **Consuming Spec Feedback:** The Merger Agent reviews any `spec_analysis.txt` files generated by the Implementation Agents (which document ambiguities, contradictions, or external incompatibilities).
   - **Spec Evolution:** If there is any ambiguity, contradiction, or omission in the Butler spec itself, or if a `spec_analysis.txt` suggests a streamlining recommendation, the Merger Agent makes recommended changes to the specifications under the `spec/` directory or modifies the `UPDATE.md` skill on the merger branch to resolve the issue cleanly and elegantly.
   - **Artifacts:** Only specification/skill changes (under `spec/` or `UPDATE.md`), the implementation feedback (`merge_analysis.txt` additions/deletions), and the resolution of `spec_analysis.txt` files are committed and pushed.

2. **Implement Spec (The Implementation Agent - governed by `UPDATE.md`):**
   - **Responsibility:** Responsible for modifying code inside its specific implementation directory (`butler/` and `bin/`) to satisfy the specification, resolving identified non-compliance, and raising spec-level feedback.
   - **Execution Loop:** Upon checkout, the Implementation Agent merges in the latest specs from `main`, checks for `merge_analysis.txt`, and audits the specification files under `spec/`.
   - **Feedback Loop to Spec:** If there are specific ambiguities, contradictions, omissions, or external spec (e.g. UUFI, libraries) incompatibilities/inconsistencies that prevent creating a clear, correct implementation, the Implementation Agent MUST create or update a `spec_analysis.txt` file at the root of the repository. This file must be highly precise, describing clear, non-speculative problems and providing elegant recommendations to streamline the spec (aiming for an elegant, single standard way of doing things, avoiding workarounds or slop). Performance improvements, speculative ideas, or security enhancements should NOT be listed. If no such obstacles exist and the spec is clear and correct, any existing `spec_analysis.txt` file MUST be removed.
   - **Code Refinement:** The agent refines the implementation logic until all requirements in `spec/` and all issues reported in `merge_analysis.txt` are fully resolved and local smoke tests pass. Once verified, `merge_analysis.txt` is deleted, and the updated implementation is committed and pushed.
   - **Spec Integrity:** The Implementation Agent never modifies specifications or files outside of the allowed directories (`butler/`, `bin/`, and `UPDATE.md` if directed).

Through this continuous loop of verification and feedback (Merger Agent) and refinement and spec-auditing (Implementation Agent), the Butler specification is iteratively audited, streamlined, and hardened, achieving true specification-driven robustness.

## Pre-Requisites & Validation

1. **Branch Verification:**
   - This procedure MUST only be applied to branches that start with `impl_` (e.g., `impl_B`).
   - This procedure MUST NOT be performed on the `main` or `merger` branch, or any other unrecognized branch.
   - Verify that the active git branch name is valid and the working directory is clean before beginning.

2. **Upstream Synchronization:**
   - Fetch the latest changes from the remote repository.
   - Merge `origin/main` into the current branch to pull in any specification updates.
   - *Conflict Resolution:* If merge conflicts occur within files in `spec/`, since they are immutable, the agent MUST always auto-resolve them in favor of `origin/main` without exception, even if it breaks the current branch's historical custom tests, as specs on `main` are the immutable source of truth.

3. **Automatic Port Status Check:**
   - Prior to launching any local brokers or executing tests, the update process MUST run a dynamic pre-check to detect if there are any active processes currently listening on the branch-mapped MQTT/testing port (such as `40000-49999`) or standard etcd/MQTT ports.
   - This check can be performed using standard utilities (such as `ss -lntp`, `netstat`, or `lsof`) or programmatic socket connection attempts.
   - If an active broker or process is detected on the target port outside of the test runner's orchestration, the runner MUST list the active process info and PID (if accessible) to standard error to assist in diagnosing and debugging rogue or manually started brokers before proceeding.

## Code Compliance, Spec Auditing & Bidirectional Feedback

To ensure the codebase complies with any new or modified specifications, resolves interoperability failures, and raises precise specification-level feedback:

1. **Audit Spec Changes and Merger Feedback:**
   - Identify which specification files changed during the merge by running a differences check on the `spec/` directory.
   - Carefully review changes in `spec/butler.md`, `spec/blobstore.md`, and `spec/update.md`.
   - **Check for Merger Agent Feedback:** Check if `merge_analysis.txt` exists at the root of the repository. This file is provided by the merger agent (via the `MERGER.md` skill) and contains detailed diagnostic reports on implementation-specific failures and non-compliance uncovered during cross-implementation testing.

2. **Audit the Specification and Generate Spec Analysis:**
   - **Raise Spec-Level Issues:** If there are specific ambiguities, contradictions, omissions, or external spec (e.g. UUFI, external libraries) incompatibilities/inconsistencies that prevent creating a clear and correct implementation, the agent MUST create or update a `spec_analysis.txt` file at the root of the repository.
   - **Content Guidelines:** This file must contain only clear, non-speculative, and actionable problems along with clear recommendations and guidance aiming for an elegant and streamlined specification. Do not include performance improvements, security enhancements, speculative thoughts, or exploratory ideas. Do not try to "just make things work" by allowing multiple redundant ways of doing the same thing; aim for a single, streamlined solution.
   - **Removal Condition:** If there are no clear problems or ambiguities in the specification preventing a clean and correct implementation, the `spec_analysis.txt` file MUST be removed.

3. **Update Implementation Logic:**
   - Symmetrically update implementation code in `butler/` and command wrappers in `bin/` to satisfy all newly introduced specification requirements and resolve the exact failures detailed in `merge_analysis.txt`.
   - Once all failures documented in `merge_analysis.txt` have been resolved and verified to pass the smoke tests, remove the `merge_analysis.txt` file from the repository to signal that the implementation is clean, fully compliant, and ready to be re-verified by the merger agent.

## Verification & Testing Workflow

Before finishing, the implementation MUST be verified to prevent regressions:

1. **Environment Setup:**
   - **Environment Validation:** Prior to setting up the environment, verify that Python virtual environment and dependencies are satisfied according to the authoritative validation steps specified in `spec/butler.md` Section 10.1.1.
   - Run the local setup utility to ensure the MQTT broker or other local infrastructure is running and the connection specifications are correct.
   - **Explicit Broker Invocation:** The local UDMI broker startup utility (`impl/udmi/bin/start_local`) MUST be invoked directly by passing the local site model directory path (e.g., `testing/udmi_site_model`) as its first command-line argument, and the connectivity specification in standard UDMI compatible format (specifically `//mqtt/localhost:$mqtt_port` or `mqtt/localhost:$mqtt_port`) as its second command-line argument:
     ```bash
     impl/udmi/bin/start_local testing/udmi_site_model //mqtt/localhost:$mqtt_port
     ```
     This format is required to support the custom bash-based parsing of `start_local` without overrides or hacks. Tools and update scripts MUST NOT inspect, read, or parse the internal contents of `start_local` or other local setup utilities to determine port configurations; they must rely strictly on passing these arguments directly on the command line.
   - **Non-Privileged/Non-Sudo Execution:** To run in isolated, sandboxed environments without root privileges, all tools and background scripts MUST automatically operate entirely in non-privileged (no-sudo) user-space whenever a specific, non-default port is configured or passed, completely bypassing any `sudo` or `systemctl` commands. To ensure realistic validation of the system's automatic no-sudo user-space execution paths, any test harness, runner script, or supporting tool MUST NOT explicitly set, export, or pass the `UDMI_NO_SUDO` environment variable during testing, as it is an internal-only implementation detail of UDMI. Under no circumstances should any `sudo` simulation wrapper or privilege elevation bypass be used. If any `sudo` or privilege elevation is still called, required, or executed, the system MUST immediately flag it as a fundamental compatibility error and fail-fast.
   - **System Package Integrity & No Library-Path Hijacking:** All background service daemons (such as `mosquitto`, `etcd`, `java`) and tools under test MUST rely strictly on the standard system libraries and configurations natively provided by the host OS or container environment. Modifying, overriding, or prepending directory paths to environment variables like `LD_LIBRARY_PATH`, `DYLD_LIBRARY_PATH`, or `PATH` to inject loose, stashed, or dynamic shared libraries (such as `.so`, `.dylib`, or `.dll` files) from within the repository is strictly PROHIBITED. If an executable or dependency fails to load due to a missing system library, the test runner or agent MUST NOT attempt to dynamically resolve it using local library stuffing or environment hacks. Instead, it MUST immediately fail-fast and report that the host environment lacks the required system packages, allowing the underlying environment to be fixed properly.
   - **Strict Non-Interactive Fail-Fast Environment:** All environment setups, tool executions, and test runners MUST be executed inside a strictly non-interactive environment with Standard Input (`stdin`) closed or redirected to `/dev/null` (e.g., `cmd < /dev/null`). Environment variables disabling prompts (e.g., `CLOUDSDK_CORE_DISABLE_PROMPTS=1`, `DEBIAN_FRONTEND=noninteractive`, and disabling `sudo` asks) MUST be exported to guarantee that any process attempting to prompt for user input or credentials immediately crashes and fails-fast.
   - **Predictable Workspace Scaffolding Mandate:** The directory structure of the testing workspace is fixed, immutable, and strictly defined. The agent MUST NOT execute generic directory scans, recursive finds, or path explorations to determine the layout. Every sibling implementation directory `impl/{ID}/` MUST conform to this exact relative layout: local setup script must be at `impl/{ID}/bin/setup`, local test execution script must be at `impl/{ID}/bin/smokeit`, local immutable UDMI dependency must be at `impl/{ID}/impl/udmi/`, and local isolated site model must be at `impl/{ID}/testing/udmi_site_model/`. Any deviation from this structure is a critical structural violation. If any path is missing or non-compliant, the test runner MUST immediately hard-fail.
   - **Opaque "Black-Box" Tooling Rule:** All dependencies, scripts, and utilities inside the immutable `impl/udmi` repository MUST be treated strictly as black boxes. Opening, parsing, reading, or analyzing the source code of any UDMI utility (such as `start_local`, `start_mosquitto`, `start_udmis`, `setup_ca`, etc.) to understand or debug its internal logic is strictly PROHIBITED. The tools must either run successfully according to their command-line interfaces or fail. If a tool crashes or behaves unexpectedly, the agent MUST NOT attempt to diagnose, trace, or work around the internal script logic; it MUST immediately declare a tooling error and hard-fail.
   - **Opaque Setup Responsibility:** The test harness MUST NOT perform manual environment preparation steps on behalf of the implementations, such as copying files, creating symbolic links, stashing certificates, or configuring dynamic port overrides. All environment preparation, cloning of internal site models, virtual environment activation, and local service boots MUST be handled natively and opaque-ly inside the implementation's own `bin/setup` script. The harness simply invokes `bin/setup <conn_spec>` under the fail-fast timeout, and if the setup does not exit with code `0`, it is a hard test failure.
   - **Shared Port Prohibition:** To allow simultaneous test runs on the same machine, sharing default MQTT ports (e.g., `1883`, `8883`) or etcd ports is strictly prohibited. The system MUST dynamically allocate or negotiate isolated, non-default ports for the local MQTT broker and etcd instances as specified by the SHA256 branch hashing port mapping formula in `spec/butler.md` Section 10.
   - *No Diagnosis of UDMIS Errors:* The system MUST NOT attempt to diagnose or fix any potential errors with UDMIS. Specifically, when starting the broker or DUT, it should either work as specified or report an error. The system should either start up and work as intended, or fail with a clear error indicating that the UDMIS specification or system is broken.
   - *Blobstore Provider Configuration:* During this automated update flow, the system must default to a local provider. It is acceptable to specify a non-local `BUTLER_BLOBSTORE_PROVIDER` (such as `gcs`) ONLY if it is explicitly configured via environment variables and all necessary authentication and cloud resources are already set up.

2. **Execute Code Quality & Unit Tests:**
   - Prior to running integration smoke tests, the update flow MUST execute any configured static code analysis tools, linters, or standard unit tests if they are configured in the repository to verify code health.

3. **Execute Smoke Tests:**
   - Run the automated, non-interactive integration smoke tests (`bin/smokeit`) to verify the complete registration, update, and failure rollback logic.
   - *Handling Failures:* If tests fail, diagnose using log outputs, apply necessary bug fixes strictly within the `butler/` and `bin/` directories, and re-run until all tests pass.
   - *Test Outputs and Logs:* Verifier and smoke-test outputs (e.g., test logs or summary files) SHOULD be saved locally in an ignored directory (such as `testing/` or `tmp/`) for debugging, but they MUST NOT be committed to the repository or uploaded as part of the PR/commit artifact.

## File and Workspace Integrity Audit

1. **Strict File Restrictions:**
   - At the end of the run, the *only* allowed changes in the repository are to files in the `butler/` and `bin/` directories, the removal/update of `merge_analysis.txt`, the creation, update, or removal of `spec_analysis.txt`, and changes to `UPDATE.md` itself when explicitly directed.
   - There should NOT be any other stray files or changes made to any other files (e.g., `spec/`, `.gitignore`, `AGENTS.md`, `WORKFLOW.md`, etc.).
   - It is acceptable to have temporary files in directories ignored by `.gitignore` (such as `testing/` or `tmp/`), but the `.gitignore` file itself MUST NOT be modified. Any generated verifier, smoke test logs, or summary files MUST NOT be committed or tracked.
   - **Repository and Workspace Cleanliness:** All runtime-created operational folders, log directories, virtual environments, custom overlays, and script outputs (including `var/`, `etc/`, `venv/`, `.venv/`) MUST be nested exclusively inside pre-defined ignored locations (specifically within the `tmp/` or `testing/` directories). Creating these runtime files at the repository root or any non-ignored directories is strictly prohibited to prevent repository pollution and untracked changes.

2. **Commit and Push Procedure:**
   - Stage and commit only the allowed changed files (do not use global or untargeted add commands unprompted).
   - If no functional code changes were required to comply with the specs, create an empty commit with a simple "No changes required" message,
     with no other text, in the git commit log to signal the successful run.
   - If changes were made, propose a clear, concise commit message explaining the updates. The system should only summarize *changes* that were actually made and should not describe what tests or checks were performed.
   - Push the committed branch to the remote repository.

## Testing Encountered Issues

1. **Actually Encountered Issues**
   - **Encountered Issues List:** When actually testing and verifying the implementation under this procedure, the agent or runner MUST dynamically list and output only the specific operational issues, errors, or bugs that were actually encountered due to a bad specification and resolved *during that specific run* of `UPDATE.md`. It MUST NOT report any theoretical or static-analysis-based issues, nor any issues that were statically pre-instructed or requested to change, ensuring the documented issues are strictly empirical runtime findings from that active execution. It should NOT report any errors from a bad _implementation_, rather only errors resulting from a bad _specification_ (input `.md` file).
2. **Avoid Hacky Workarounds**
  - **Should not hack:** If there is a problem identified with UDMI or UUFI that is in conflict with the Butler spec the system should NOT work around the problem in violation of the spec. Rather, it should just report the error and fail. The system should report and hard fail any fundamental specification or tooling incompatibilities (including any requirement for `sudo` or privilege elevation, which must never be worked around with mock `sudo` wrappers) rather than trying to work around them.
